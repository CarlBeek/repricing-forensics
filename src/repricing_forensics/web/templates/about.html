About this analysis

What is EIP-7904?

EIP-7904 proposes raising the gas cost of eight EVM opcodes whose prices have not been updated since Ethereum's Frontier release in 2015. Over that time, improvements to EVM interpreters have shifted the relative cost profile of instructions, but the gas schedule has not kept pace. Opcodes like DIV, SDIV, MOD, SMOD, ADDMOD, MULMOD, EXP, and KECCAK256 now consume more real CPU time per unit of gas than their peers, creating a potential vector for denial-of-service attacks that exploit the mispricing.

The proposed changes also adjust precompile costs to keep them consistent with the repriced opcode schedule. The full specification is available in EIP-7904.

How we measure impact

We forked reth and modified its EVM gas table to reflect the proposed schedule. The fork re-executes historical mainnet blocks exactly as they originally ran -- same state, same transaction ordering, same block headers -- but charges the new gas prices. After each transaction, we compare the baseline execution result (status, gas used, logs) against the repriced result and record every difference.

Re-execution happens at the EVM level, not via tracing. The modified node processes each opcode natively, so the measurements capture the actual gas accounting path rather than an after-the-fact estimate.

What the data captures

For every transaction in the sample, the analysis records:

  • Baseline gas used -- gas consumed under the current schedule.
  • Schedule gas used -- gas consumed under the EIP-7904 schedule.
  • Status change -- whether the transaction flips from success to revert (or vice versa).
  • Divergence location -- the program counter and call depth where execution first diverges.
  • Call stack snapshot -- the full call chain at the point of divergence, including contract addresses and function selectors.
  • Per-opcode deltas -- how many additional gas units each repriced opcode contributes in the affected transaction.

Wallet-fixable vs contract-broken

Not all failing transactions represent real breakage. If a transaction runs out of gas only because the user-supplied gas limit was too low, a wallet can fix the problem by raising the limit. We call these wallet-fixable transactions. They fail under the new schedule but would succeed if re-submitted with a higher gas limit.

Contract-broken transactions are different. These fail because a smart contract hardcodes a gas value in an internal call -- for example, passing a fixed gas parameter to CALL. No wallet adjustment can help because the contract itself constrains the gas available to the sub-call. Fixing these requires a contract upgrade or migration.

The classifier uses a depth heuristic: if the out-of-gas event occurs at call depth greater than 1 and the parent call forwarded a fixed (non-63/64) amount of gas, the transaction is marked contract-broken. Otherwise it is marked wallet-fixable.

The SSTORE refund complication

EVM gas accounting distinguishes between pre-refund and post-refund gas usage. SSTORE operations that clear storage slots earn a gas refund, but this refund is applied after execution completes. When a transaction runs out of gas under the new schedule, no refund is issued (the transaction reverted), so the reported gas_used can be dramatically higher than the baseline -- not because the opcodes cost that much more, but because the refund that would have offset the cost was never applied.

This means the raw gas delta for OOG transactions inflates the apparent cost increase. The analysis reports both the gross delta and the pre-refund headroom to give a more accurate picture.

The 63/64 rule and deep call chains

When a contract makes an external call, EIP-150 requires it to withhold 1/64 of available gas (the "63/64 rule"). The callee receives at most 63/64 of the caller's remaining gas. In deep call chains this compounds: each level withholds a fraction, and by depth 10 or more, a significant share of the original gas has been retained across the stack.

Under higher opcode prices, the compounding effect is amplified. Contracts that forwarded gas via CALL without specifying an explicit limit relied on the 63/64 share being "enough." When per-opcode costs rise, the same 63/64 share may no longer cover the callee's execution. Contracts that hardcode a specific gas stipend (e.g., 2300 for the transfer stipend) are especially fragile because the stipend was sized for the original gas schedule.

Limitations

  • Identical state assumption. Re-execution replays transactions against the same state that existed when they originally ran. In practice, if gas costs change, earlier transactions in a block may produce different state, which could alter the outcome of later transactions. This second-order effect is not captured.
  • Heuristic classification. The wallet-fixable vs contract-broken distinction relies on a depth-and-gas heuristic. Edge cases (e.g., a contract that forwards gasleft() - 100) may be misclassified.
  • Labeling coverage. Contract and project labels come from public sources (Etherscan, 4byte directory). Unlabeled contracts appear by address only.
  • Sample, not census. The analysis covers a specific range of mainnet blocks. Contracts deployed after the sample period or rarely called within it may not appear in the results.
